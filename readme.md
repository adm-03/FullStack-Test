# Fullstack Презентация Разработчика с AI-интеграцией (FastAPI + Docker)

Интерактивный лендинг-портфолио со встроенной формой обратной связи и кнопкой генерации случайных мыслей «Вдохновение дня» на базе генеративного ИИ. 

Проект выполнен в рамках тестового задания и демонстрирует полный цикл разработки: от адаптивной валидной верстки до контейнеризированного асинхронного API, спроектированного по enterprise-стандартам.

## 🛠 Стек технологий

*   **Frontend:** HTML5 (валидная, семантическая верстка), CSS3 / SCSS (адаптивная сетка), Vanilla JavaScript (ES6+, асинхронное управление DOM).
*   **Backend:** Python 3.12, FastAPI, Pydantic V2 (валидация данных), Pydantic Settings (управление конфигурацией окружения).
*   **AI-Интеграция:** Groq Официальный Python SDK (асинхронный клиент, модель `llama-3.1-8b-instant`).
*   **Инфраструктура:** Docker, Docker Compose, Пакетный менеджер `uv` (Astral).
*   **Почтовый шлюз:** `aiosmtplib` (асинхронный SMTP-клиент).

## ⚙️ Локальный запуск (Docker Compose)

Проект полностью контейнеризирован и разворачивается одной командой:

1. Склонируйте репозиторий и перейдите в корень проекта.
2. Создайте файл секретов бэкенда на основе примера:
   ```bash
   cp backend/.env.example backend/.env
   ```
   *(Заполните ваши реальные ключи `GROQ_API_KEY` и `SMTP_PASSWORD` внешнего приложения почты).*
3. Запустите сборку и старт инфраструктуры:
   ```bash
   docker compose up --build
   ```
4. Бэкенд поднимется на порту `8000`. Интерактивная документация Swagger доступна по адресу: `http://localhost:8000/docs`.
5. Фронтенд можно запустить локально через VS Code расширение **Live Server** либо легковесный сервер Python из папки `frontend` (`python -m http.server 3000`).

## 🦾 Подробная реализация формы обратной связи

Работа формы обратной связи построена на принципе **асинхронного неблокирующего fullstack-цикла** и разделена на три изолированных этапа для обеспечения максимальной производительности интерфейса и отказоустойчивости системы:

### Этап 1. Фронтенд и UI/UX (Клиентская часть)
*   **Сбор данных и пре-валидация:** Когда пользователь нажимает кнопку «Отправить», JavaScript перехватывает событие (`e.preventDefault()`), собирает значения полей и запускает валидацию на регулярных выражениях (`emailRegex` и `phoneRegex`).
*   **Loading-состояние:** Если данные введены корректно, JS-скрипт блокирует кнопку отправки (`submitBtn.disabled = true`), предотвращая повторные случайные клики (защита от спама), и выводит на экран анимированный лоадер.
*   **Отправка данных:** Данные упаковываются в JSON-объект. На фронтенде инпут комментария называется `message`, но скрипт пересобирает его в ключ `comment`, чтобы он строго соответствовал требованиям Pydantic-модели бэкенда. Запрос улетает через асинхронный `fetch` на порт бэкенда.

### Этап 2. Бэкенд и Сквозная Валидация (API-слой)
*   **Входной контроль:** Запрос принимает роутер эндпоинта `/api/send_mail`. Данные мгновенно попадают в Pydantic-модель `MailSchema`.
*   **Жесткая проверка:** Поле `email` проверяется встроенным типом `EmailStr` (проверяется синтаксис, наличие `@` и существование домена), а поле `phone` валидируется регулярным выражением, идентичным фронтенду. Если запрос отправлен в обход сайта (например, через Postman/скрипт) с некорректными данными, Pydantic заблокирует его и выбросит ошибку `422 Unprocessable Entity`.
*   **Мгновенный ответ (Паттерн неблокирующего ввода-вывода):** Если данные валидны, бэкенд **НЕ начинает** сразу отправлять письма. Он регистрирует задачу в фоновом потоке и **мгновенно возвращает фронтенду HTTP-статус `202 Accepted`**.

### Этап 3. Фоновое выполнение и Обработка ошибок (Слой Бизнес-логики)
*   **Освобождение клиента:** Как только фронтенд получает статус `202`, на сайте мгновенно выключается лоадер и включается `success состояние` («Успешно! Заявка принята»). Пользователь свободен, интерфейс разблокирован.
*   **Работа в фоне (`BackgroundTasks`):** В этот момент на бэкенде в фоновом потоке вызывается бизнес-логика — класс `SendMailUseCase` и его метод `execute()`.
*   **Асинхронная SMTP-отправка:** Внутри юзкейса с помощью библиотеки `aiosmtplib` открывается асинхронная зашифрованная SSL-сессия (порт 465) с сервером Mail.ru. Бэкенд последовательно формирует и отправляет два разных письма: уведомление для владельца сайта со всеми логами и подтверждение для самого пользователя, что его обращение принято в работу.
*   **Резервное логирование (Error-состояние):** Вся фоновая отправка обернута в блок `try/except`. Если на стороне почтового провайдера произойдет сбой, антифрод-система сбросит сессию или пропадет интернет, код перехватит эту ошибку и запишет её в системные логи Docker-контейнера через `logger.error()`. При этом интерфейс сайта никак не пострадает и не зависнет.

### Ключевые архитектурные тезисы реализации:
1.  **Dual-layer Validation:** Валидация работает и на клиенте (для UX), и на сервере (для безопасности).
2.  **Мгновенный статус 202:** Сайт работает молниеносно, потому что бэк отдает ответ до начала физической отправки писем.
3.  **Фоновые задачи (`BackgroundTasks`):** Тяжелая сетевая работа с SMTP полностью изолирована в фоне и застрахована блоками `try/except` от падения приложения.

## Интеграция с AI
*   Генерация завязана на высокопроизводительную платформу **Groq LPU** через их официальный асинхронный Python SDK (`from groq import AsyncGroq`).
*   Ресурс клиента инициализируется и закрывается контекстным менеджером **`lifespan`** в `main.py`, защищая операционную систему контейнера от утечки сетевых сокетов.
*   Передача клиента в слой `usecases` происходит через **Dependency Injection** (`Depends`), изолируя роуты от деталей реализации работы ИИ.
*   Внедрен паттерн **Fallback (отказоустойчивость)**: при возникновении сетевых коллизий или лимитов со стороны внешнего API, блок `except` логирует ошибку и отдает пользователю валидную фразу из встроенного локального пула, обеспечивая **100% аптайм UI**.

## 🤖 Отчет об использовании AI-инструментов в работе

В соответствии с требованиями технического задания, в процессе проектирования и разработки проекта активно применялись AI-инструменты. Искусственный интеллект использовался в роли продвинутого парного программиста (Pair Programmer) для ускорения рутинных операций, в то время как ключевая архитектура и системная логика создавались и контролировались вручную.

### 1. Что делегировалось AI (GitHub Copilot)
*   **Генерация UI-слоя:** GitHub Copilot использовался для ускорения написания семантической HTML5-разметки лендинга и генерации базовой структуры SCSS-компонентов (сеток, адаптивных медиа-запросов и переменных оформления).
*   **Рутинные конструкции:** ИИ привлекался для первичного написания комплексного регулярного выражения валидации телефонов и подготовки шаблонных конструкций (типизированных конфигов ответов для Swagger).

### 2. Что выполнялось исключительно вручную (Проектирование и Логика)
*   **Архитектурный дизайн:** Разбиение бэкенда на изолированные слои по канонам *Clean Architecture* (выделение Use Cases / Вариантов использования, вынос конфигурации в Pydantic Settings, создание независимого слоя DTO/Schemas).
*   **Инфраструктурный сетап:** Написание оптимизированного двухэтапного Dockerfile (*Multi-stage build*), настройка локального кэширования зависимостей через директивы `--mount=type=cache`, интеграция пакетного менеджера `uv` и перевод сборки на внутреннее российское зеркало Яндекса (`pypi.yandex.ru`) для обхода сетевых тайм-аутов.
*   **Асинхронное управление ресурсами:** Инициализация и менеджмент сессий клиентов `AsyncGroq` и `aiosmtplib` внутри контекстного менеджера `lifespan` FastAPI для предотвращения утечек сокетов под нагрузкой.
*   **Отказоустойчивость (Fallback паттерн):** Проектирование логики перехвата ошибок внешних API (ошибок авторизации, геоблокировок РФ и сетевых коллизий), позволяющей бэкенду бесшовно отдавать клиенту данные из локального пула заглушек без падения интерфейса.
*   **Бэкэнд:** Писался руками.
### 3. Ручное код-ревью и исправление ошибок ИИ
В ходе интеграции сгенерированного ИИ контента было проведено глубокое ручное код-ревью, в результате которого устранены следующие критические расхождения:
1.  **Синхронизация контрактов (Data Mapping):** ИИ сгенерировал форму с полем `message`, тогда как Pydantic-схема бэкенда строго ожидала поле `comment`. Несоответствие было обнаружено на этапе интеграционного тестирования и исправлено вручную пересборкой payload в JS-обработчике во избежание массовых ошибок `422`.
2.  **Разрешение конфликтов PYTHONPATH:** При вынесении исходного кода в изолированную директорию `src/` интерпретатор падал с ошибкой `ModuleNotFoundError: No module named 'backend'`. Проблема решена ручным перестроением абсолютных импортов от корня `src` и явной кастомизацией команды запуска Uvicorn в Docker через флаг `--app-dir src`.
3.  **Корректировка версий моделей:** Сгенерированная ИИ конфигурация для Groq вызывала модель `llama3-8b-8192`, которая на момент деплоя была выведена провайдером из эксплуатации (Decommissioned). Вручную проведена миграция на актуальный и поддерживаемый инференс `llama-3.1-8b-instant`.
4.  **Кривые Dockerfile и конфликты с региональными ограничениями**
